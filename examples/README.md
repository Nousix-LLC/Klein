# Klein examples

A curated set of idiomatic Klein programs that exercise the language end-to-end,
each committed alongside a **generated** golden output. These are the shared
example/golden data set the end-to-end test suite (`tests/integration/**`,
`tests/errors/**`) consumes ‚Äî it is data-driven from [`index.json`](./index.json)
and never hard-codes per-example specifics.

Run any program with the shipped CLI:

```sh
klein examples/fibonacci.kl
# or, from a clone without a global install:
node bin/klein.mjs examples/fibonacci.kl
```

Klein programs emit output **only** through the `print` / `println` builtins ‚Äî
the CLI does not echo a program's final value, so every example that shows
output calls `println`.

## The files

For each example `<name>`:

| File          | Meaning                                                                                                          |
| ------------- | ---------------------------------------------------------------------------------------------------------------- |
| `<name>.kl`   | the Klein program                                                                                                |
| `<name>.out`  | **golden stdout** ‚Äî the exact bytes the program prints, color disabled (may be empty)                            |
| `<name>.diag` | **golden diagnostics** ‚Äî the exact rendered snippet+caret stderr, `NO_COLOR` (present only for `error` examples) |
| `index.json`  | machine-readable manifest: `{ name, kind, expectedCodes? }` per example                                          |

`kind` is `"ok"` (runs clean, exit `0`, no diagnostics) or `"error"` (produces
one or more diagnostics, exit `1`). For `error` examples, `expectedCodes` is the
ordered list of structured [`ErrorCode`](../contracts/errors.ts) string values
the run emits, in source order ‚Äî tests key off these codes, never on message
text.

## The set

### `ok` ‚Äî clean runs

| Example                                      | Demonstrates                                                                                        |
| -------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| [`fibonacci.kl`](./fibonacci.kl)             | recursion, an iterative variant, `for` loops, first-class functions                                 |
| [`fizzbuzz.kl`](./fizzbuzz.kl)               | `for` loops, modulo, `if` / `else if` / `else` chains                                               |
| [`closures.kl`](./closures.kl)               | `fn` expressions closing over their defining scope; private per-instance state; partial application |
| [`higher_order.kl`](./higher_order.kl)       | `range`, `map`, `filter`, `reduce`, `sort` (with a user comparator), `join`                         |
| [`data_structures.kl`](./data_structures.kl) | insertion-ordered objects, `keys`/`values`/`has`, reference arrays, `push`/`pop`, nesting           |
| [`strings.kl`](./strings.kl)                 | `trim`/`upper`/`lower`, `split`/`join`, `chars`, `ord`/`chr`, `contains`, `slice`                   |

### `error` ‚Äî intentional faults for the diagnostics demo

Each error example contains a deliberate fault so the source-anchored diagnostic
renderer can be demonstrated (and asserted against).

| Example                                        | Fault                                                                    | Code(s)          |
| ---------------------------------------------- | ------------------------------------------------------------------------ | ---------------- |
| [`errors_undefined.kl`](./errors_undefined.kl) | assigning to a name never declared with `let`                            | `E3001`          |
| [`errors_type.kl`](./errors_type.kl)           | `1 + "two"` ‚Äî `+` needs two numbers or two strings (no coercion)         | `E3002`          |
| [`errors_divzero.kl`](./errors_divzero.kl)     | division by zero inside a function (diagnostic carries a call stack)     | `E3009`          |
| [`errors_syntax.kl`](./errors_syntax.kl)       | two syntax faults in one file ‚Äî the parser recovers and reports **both** | `E2003`, `E2003` |

## How the goldens were generated

Every `.out` / `.diag` is **generated by running the real interpreter**, never
hand-authored, so a golden can never disagree with real behavior. Each program
is driven through the `interpret()` facade (`src/index.ts`) ‚Äî the exact code path
the CLI's runner uses ‚Äî capturing the `print`/`println` sink as `.out` and, for
error examples, rendering each diagnostic via
`formatDiagnostic(d, source, { color: false })` (one per line, as the CLI writes
them to stderr) as `.diag`. `expectedCodes` is read directly from
`interpret().diagnostics` in source order.

Diagnostics are generated with the source name `examples/<name>.kl` (the
repo-root-relative path), so the goldens are identical whether reproduced
in-process via `interpret(source, { sourceName: "examples/<name>.kl" })` or by
spawning the shipped executable from the project root:

```sh
NO_COLOR=1 node bin/klein.mjs examples/errors_type.kl
```

All ten goldens have been confirmed byte-for-byte against that real executable.
